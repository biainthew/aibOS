정보를 안전하게 전송하기 위해 JSON 객체를 사용하는 토큰 기반 인증 방식


### 토큰의 구조


JWT는 점을 구분자로 세 부분으로 나뉨

- **Header** : 토큰의 타입(JWT), 사용 중인 암호화 알고리즘(RSA, HS256 등) 정보가 담겨 있음
- **Payload** : 실제 유저의 정보(ID, 유효기간)가 담겨 있음, 이 데이터를 Claim 이라고 부름
- **Signature** : 헤더와 페이로드를 합친 뒤 서버만 아는 Secret Key 로 암호화 한 값, 토큰이 중간에 조작되지 않았음을 증명

### 인코딩과 보안


헤더와 페이로드는 누구나 읽을 수 있는 Base64 방식으로 인코딩되어있음


별도의 암호화가 아니기 때문에 누구나 복호화해서 내용을 볼 수 있음 따라서 비밀번호 같은 민감한 정보는 넣으면 안 됨


### 암호화 방식

- **HS256(대칭키)** : 하나의 Secret Key 로 암호화하고 복호화함
- **RS256(비대칭키)** : 서버가 개인키로 서명, 검증하려는 쪽은 공개키로 확인

### 왜 JWT를 쓰는지 ?

- 서버가 세션을 유지할 필요 없이 클라이언트가 보낸 토큰의 시그니처만 확인하면 됨
- 서버 1 에서 발행한 토큰이라도 서버 2가 동일한 Secret Key만 가지고 있다면 클라이언트가 어느 서버로 요청을 보내든 정상적으로 인증됨
- Redis 같은 별도의 세션 서버를 두지 않아도 서버를 무한히 늘릴 수 있는 장점이 있음

### JWT 와 Redis 를 같이 쓰는 이유는 ?


JWT는 서버가 상태를 저장하지 않기 때문에 토큰을 누군가 훔쳐가도 서버가 그 토큰을 실시간으로 무효화 할 방법이 없음 -> 유효기간이 끝날 때까지 도둑이 내 계정 맘대로 쓰기 가능


이걸 해결하기 위해 토큰을 두 종류로 나눔


**Access Token (JWT)**

- 유효기간이 짧음
- 실제 API 요청 보낼 때 사용
- 서버는 이걸 보고 바로 인증해줌
- 자바스크립트 내부 변수나 로컬스토리지에 저장해둠

**Refresh Token (JWT)**

- 유효기간이 김
- Access Token 이 만료되었을 때 새로 발급받기 위한 용도
- HttpOnly 쿠키에 저장해둠
- 이 토큰을 Redis 에 저장

**Redis 를 쓰면 얻는 이득**

- 유저가 로그아웃 하면 Redis 에 있는 Refresh Token 을 삭제하면 됨
- 새로운 기기에서 로그인 할 때 기존 Redis Token 을 덮어쓰면 이전 기기에서의 세션을 끊을 수 있음
- Redis 는 메모리 기반이라 DB보다 훨씬 빠르게 토큰 유효성을 체크할 수 있음
