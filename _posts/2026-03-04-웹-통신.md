JWT 이해하기 전 알아볼 개념들


### 1. 세션


**세션 기반 인증의 기본 흐름**

1. 클라이언트가 사이트에 접속 시 서버 세션에 Session ID 가 생성, 저장
2. 유저가 로그인을 시도하면 서버는 DB에서 정보를 확인
3. 인증 성공 시 서버는 해당 Session ID 에 유저 정보를 연결하여 저장
4. 서버는 클라이언트에게 Session ID 를 담은 쿠키를 리턴

**멀티 서버 환경의 문제점**


서버가 여러 대인 환경에서는 로드밸런서가 요청을 분산 처리


클라이언트가 초기 접속 시 A 서버에서 세션을 생성했으나 로그인 요청이 B 서버로 전달될 경우 B 서버 세션에는 해당 유저의 정보가 없는 문제 발생


-> 사용자는 로그인을 했음에도 불구하고 인증이 풀리거나 오류 발생


**해결 : 외부 세션 스토리지**


문제 해결을 위해 서버 내부 메모리가 아닌 모든 서버가 공유하는 공동 저장소를 사용


    **관계형 DB**

    - 하드디스크 기반 저장 방식
    - 데이터 입출력(I/O) 속도가 느려 병목현상 발생 가능성 높음

    **In-Memory DB (Redis 등)**

    - 데이터를 RAM에 저장하는 메모리 공유 서버
    - 디스크를 거치지 않고 메모리에서 바로 값을 찾아서 속도가 빠름

### 2. TCP/UDP


**TCP**


신뢰성 중심의 연결 지향 프로토콜


사용 사례 : 웹 사이트(HTTP, HTTPS), 이메일 전송, 파일 다운로드 등


**UDP**


속도 중심의 비연결성 프로토콜


사용 사례 : 보이스톡, 전화, 동영상 스트리밍 


### 3. CIA


**기밀성**


권한이 있는 사람만 접근할 수 있어야 함


데이터 암호화, 접근 제어(ID/PW), 방화벽 등


**무결성**


정보는 정확하고 정밀하며 수정되지 않은 상태여야 함


디지털 서명, 해시함수, 버전 관리 등


**가용성**


필요할 때 서비스가 죽어 있으면 안됨


이중화, 백업 시스템, DDoS 방어 솔루션 등


### 4. RSA


**공개키로 암호화 -> 개인키로 복호화**


목적 : 데이터 보호


사례 : 비밀 메세지 전송, 키 교환 등


아무도 못 보게 하려고 암호화 한 것 


**개인키로 암호화 -> 공개키로 복호화**


목적 : 신원 증명


사례 : 공동인증서, 전자서명


내가 보낸 게 맞음을 인증하려고 암호화 한 것


### 5. RFC


인터넷 기술 표준을 정의하는 문서 시리즈


내가 배우는 HTTP, TCP, IP 등 모든 규칙의 원본은 RFC 에 있음


삼성 서버와 애플 브라우저가 통신할 수 있는 이유는 둘 다 동일한 RFC 문서를 보고 구현했기 때문


수많은 전문가의 검토를 거쳐 만들어진 규칙이므로 보안, 성능 면에서 검증된 방식
