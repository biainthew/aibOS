댓글 작성 시 게시글 작성자에게 실시간으로 알림을 전송하는 기능을 SSE(Server-Sent Events)로 구현


## SSE란 ?


서버에서 클라이언트로 단방향으로 데이터를 실시간 전송하는 기술


WebSocket과 달리 단방향 통신만 필요한 경우에 적합하고 HTTP 프로토콜을 그대로 사용하기 때문에 별도 설정 없이 적용 가능함


클라이언트에서 연결이 끊어지면 자동으로 재연결을 시도하는 기능이 내장되어 있


### **작동 흐름**

1. 브라우저가 서버에게 알림 연결 요청을 보냄
2. 서버는 `SseEmitter`  객체를 생성해서 반환하고 객체를 메모리에 보관
3. 서버에 이벤트가 발생하면 보관해둔 `emitter.send()` 를 호출해 데이터를 보냄
4. 설정한 시간이 지나거나 작업이 끝나면 `complete()` 를 호출해 연결을 닫음

> 💡 **SSE vs WebSocket vs Webhook vs Polling**  
> **SSE**  
>   
> - 특징: 서버가 클라이언트에게 일방적으로 데이터를 계속 밀어넣는 방식  
>   
> - 장점: HTTP 기반이라 가볍고 구현이 매우 쉬움 브라우저가 자동으로 재연결을 시도  
>   
> - 단점: 서버에서 클라이언트로만 보낼 수 있음  
>   
> **WebSocket**  
>   
> - 특징 : 한 번 연결하면 서버와 클라이언트가 전화를 끊지 않고 계속 대화하는것과 같음  
>   
> - 장점 : 양쪽에서 언제든 데이터를 보낼 수 있고 HTTP 헤더 오버헤드가 적어 빠름  
>   
> - 단점 : 연결 유지 자체에 자원이 많이 들고 방화벽 설정이 복잡할 수 있음  
>   
> **Webhook**  
>   
> - 특징 : 이 일이 끝나면 알림 보내라고 URL 을 남겨두는 방식  
>   
> - 장점 : 서버가 계속 기다릴 필요가 없어 자원 효율이 가장 좋음  
>   
> - 단점 : 수신측이 반드시 외부에서 접근 가능한 고정 주소를 가져야 함  
>   
> **Polling**  
>   
> - 특징 : 클라이언트가 요청을 보내면 서버가 응답을 하고 연결 종료함 그리고 다시 요청  
>   
> - 단점 : 데이터가 없어도 계속 요청을 보내니 서버와 네트워크에 부담이 큼


## 구현 흐름


클라이언트가 `/api/notifications/subscribe` 로 SSE 구독 요청을 보내면 서버는 `SseEmitter`를 생성하고 `ConcurrentHashMap`에 `memberId` 를 키로 저장함


이후 댓글이 작성되면 게시글 작성자의 `SseEmitter`를 꺼내서 알림 데이터를 전송하는 구조


> 💡 **ConcurrentHashMap 을 사용하는 이유 ?**  
> 여러 명이 동시에 데이터를 넣고 빼는 멀티스레드 환경에서는 HashMap 을 사용할 경우 데이터가 꼬이거나 무한 루프에 빠질 위험이 있음  
>   
> **Hashtable 이 있는데 왜 ConcurrentHashMap 을 사용하는지 ?**  
>   
> Hashtable 은 동기화를 위해 맵 전체에 Lock 을 걸어둠  
>   
> 그래서 한 명의 스레드가 데이터를 넣고 있으면 다른 스레드들은 그 작업이 끝날 때까지 아무 것도 하지 못하고 기다려야 해서 병목이 심함  
>   
> ConcurrentHashMap 은 버킷 단위로 쪼개서 Lock 을 걸거나 CAS(Compare And Swap) 라는 기술을 사용함 그리고 데이터를 읽는 작업에는 잠금을 걸지 않아 여러 명이 동시에 읽어도 속도가 빠름  
>   
> 따라서 웹서버 환경, 캐시 구현 등의 상황에서 사용하면 됨


## 주요 코드


**SSE 구독**


{% raw %}
```java
public SseEmitter subscribe(String email) {
    Member member = memberRepository.findByEmail(email)
            .orElseThrow(() -> new CustomException(ErrorCode.MEMBER_NOT_FOUND));

    SseEmitter emitter = new SseEmitter(60 * 60 * 1000L); // 1시간 타임아웃

    emitters.put(member.getId(), emitter);

    emitter.onCompletion(() -> emitters.remove(member.getId()));
    emitter.onTimeout(() -> emitters.remove(member.getId()));
    emitter.onError(e -> emitters.remove(member.getId()));

    // 연결 직후 더미 이벤트 전송 (연결 유지용)
    try {
        emitter.send(SseEmitter.event()
                .name("connect")
                .data("connected"));
    } catch (IOException e) {
        emitters.remove(member.getId());
    }

    return emitter;
}
```
{% endraw %}


연결 직후 더미 이벤트를 전송하는 이유는 일부 브라우저에서 첫 이벤트가 올 때까지 연결을 유지하지 않는 경우가 있기 때문


**알림 전송**


{% raw %}
```java
@Transactional
public void send(Member receiver, Member sender, Post post, NotificationType type) {
    // 자기 자신에게는 알림 전송 안 함
    if (receiver.getEmail().equals(sender.getEmail())) return;

    Notification notification = Notification.builder()
            .receiver(receiver)
            .sender(sender)
            .post(post)
            .type(type)
            .build();

    notificationRepository.save(notification);

    SseEmitter emitter = emitters.get(receiver.getId());
    if (emitter != null) {
        try {
            emitter.send(SseEmitter.event()
                    .name("notification")
                    .data(new NotificationResponseDto.Detail(notification)));
        } catch (IOException e) {
            emitters.remove(receiver.getId());
        }
    }
}
```
{% endraw %}


알림은 DB에 저장하고 SSE로도 실시간 전송함


SSE 연결이 없는 경우(오프라인 상태)에는 DB에만 저장되고 나중에 알림 목록 조회 API로 확인 가능함


## 구독 엔드포인트 설계 관련 고민


초기에는 `/api/notifications/subscribe/{memberId}` 로 경로 변수로 memberId를 받는 구조로 설계했으나 이 방식은 인증 없이도 다른 사람의 알림을 구독할 수 있는 보안 문제가 있었음


SSE는 브라우저의 `EventSource`를 사용할 때 헤더 설정이 불가능해서 토큰 인증이 어렵다는 점도 있었음


이를 해결하기 위해 `/api/notifications/subscribe` 로 변경하고 `@AuthenticationPrincipal`로 토큰에서 이메일을 추출하는 방식으로 변경함 (`fetch` API를 사용하면 헤더에 토큰을 담아서 SSE 연결이 가능하기 때문에)

## 참고

- 현재 구조는 단일 서버 기준이라 서버가 여러 대로 늘어나면 `SseEmitter`를 서버 간 공유할 수 없는 문제가 생김 → 이 경우 Redis Pub/Sub 으로 전환이 필요함
