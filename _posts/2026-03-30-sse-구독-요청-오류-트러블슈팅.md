운영 환경에서 SSE 구독 요청이 실패하는 두 가지 문제를 해결한 과정


## 문제 1 - ERR_INCOMPLETE_CHUNKED_ENCODING


### 문제 상황


로컬에서는 정상 동작하던 SSE 구독 요청이 CloudFront 배포 후 운영 환경에서 아래 에러와 함께 실패


{% raw %}
```plain text
net::ERR_INCOMPLETE_CHUNKED_ENCODING
```
{% endraw %}


### 원인

- Nginx가 기본적으로 응답을 버퍼링하기 때문
- SSE는 청크 전송 인코딩 방식으로 이벤트를 지속적으로 스트리밍하는데 Nginx가 응답을 버퍼에 쌓다가 끊어버리면서 스트림이 시작되기 전에 연결이 종료됨

    > 💡 **Nginx 의 버퍼링**  
    > - Nginx 는 기본적으로 백엔드 응답을 바로 클라이언트에게 전달하지 않고 버퍼에 모아뒀다가 한번에 보냄  
    >   
    > - 일반적인 HTTP 응답은 응답이 완성되면 한 번에 보내기 때문에 이 방식이 효율적임  
    >   
    > **SSE 의 특성**  
    >   
    > - SSE 는 일반 HTTP 응답과 달리 응답이 끝나지 않음  
    >   
    > - 연결을 열어두고 서버에서 이벤트가 생길 때마다 조금씩 데이터를 보내는 방식  
    >   
    > **문제가 생기는 이유**  
    >   
    > - Nginx 가 버퍼에 데이터를 모으려고 기다리는데 SSE 는 응답이 끝나지 않으니까 버퍼가 채워지지 않고 결국 타임아웃되거나 연결을 끊어버리면서 해당 에러 발생


### 해결

- Nginx 설정에 SSE 엔드포인트 전용 `location` 블록을 추가해서 버퍼링을 비활성화

{% raw %}
```plain text
location /api/notifications/subscribe {
    proxy_pass http://localhost:8080;
    proxy_set_header Host $host;
    proxy_set_header X-Real-IP $remote_addr;
    proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
    proxy_set_header X-Forwarded-Proto $scheme;

    proxy_buffering off;
    proxy_cache off;
    proxy_read_timeout 3600s;
    chunked_transfer_encoding on;
    proxy_set_header Connection '';
    proxy_http_version 1.1;
}
```
{% endraw %}

- 클라이언트 측에서도 `fetch + ReadableStream` 방식을 `@microsoft/fetch-event-source` 라이브러리로 교체
- 기존 방식은 SSE 전용 API가 아니라서 연결 끊김, 재연결, 인코딩 처리 등에서 불안정한 동작이 발생할 수 있었기 때문

{% raw %}
```typescript
import { fetchEventSource } from '@microsoft/fetch-event-source';

fetchEventSource(`${BASE_URL}/api/notifications/subscribe`, {
    headers: { 'Authorization': `Bearer ${token}` },
    signal: controller.signal,
    onmessage(event) {
        if (event.event === 'connect') return;
        try {
            const data = JSON.parse(event.data);
            onMessage(data);
        } catch {}
    },
    onerror(err) {
        console.error('SSE 연결 오류:', err);
    },
});
```
{% endraw %}

## 문제 2 - 새로고침 시 SSE 중복 구독


### 문제 상황

- 페이지를 새로고침할 때마다 SSE 구독 요청이 서버로 전송됨
- 새로고침을 반복하면 구독이 계속 누적되어 결국 커넥션 풀이 고갈되고 모든 API 요청이 실패함

### 원인

- SSE 구독 로직이 React `useEffect` 안에 위치해 있어서 페이지 새로고침 시 컴포넌트가 다시 마운트될 때마다 재실행됨
- 모듈 레벨의 `AbortController`가 새로고침 시 `null`로 초기화되면서 중복 연결 방지 로직을 통과하고 매번 새 연결을 생성했지만 서버에서는 이전 연결이 남아있었음

### 해결

- SSE 구독 호출을 `useEffect`에서 `handleLogin` 함수로 이동하여 로그인 시점에 딱 한 번만 구독하도록 변경
- `AbortController`를 React 컴포넌트 생명주기 외부에서 관리하는 싱글톤 `sseManager` 모듈을 도입
→ 모듈 레벨에서 관리하기 때문에 컴포넌트 리마운트와 리렌더링에 영향을 받지 않음

{% raw %}
```typescript
// src/lib/sseManager.ts
let controller: AbortController | null = null;

export const sseManager = {
    connect: (token: string, onMessage: (data: NotificationDetail) => void) => {
        if (controller) return; // 이미 연결 중이면 무시

        controller = new AbortController();
        fetchEventSource(`${BASE_URL}/api/notifications/subscribe`, {
            headers: { 'Authorization': `Bearer ${token}` },
            signal: controller.signal,
            onmessage(event) { ... },
        });
    },

    disconnect: () => {
        if (controller) {
            controller.abort();
            controller = null;
        }
    },
};
```
{% endraw %}

- 로그인 시 `sseManager.connect()` 호출, 로그아웃 시 `sseManager.disconnect()` 호출로 연결하여 생명주기를 명확하게 관리

## 참고


SSE 엔드포인트에 `proxy_read_timeout 3600s`를 설정한 이유는 Nginx 기본 타임아웃이 60초라서 60초 이상 이벤트가 없으면 연결이 끊기기 때문
